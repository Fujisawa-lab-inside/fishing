# Stage 20 S02 browser comparison

更新日: 2026-07-16

ユーザーが選択したA案に従い、S02の5時刻を直解法の基準として、ブラウザで扱うfloat32時系列との比較を作成した。追加の物理計算は行っていない。

## 比較した経路

1. S02のfloat64直接solver snapshotを、時刻ごとの基準値とした。
2. 5時刻×3成分×50,199 cellの非公開候補packへfloat32で格納した。
3. ブラウザ互換JavaScriptで各時刻を読み戻し、元の直接計算と比較した。
4. 内側3時刻を1時刻ずつ隠し、前後2時間からの直線補間で復元した。
5. 最悪だったmodel時刻-11時間について、直接計算、推定、速度ベクトル誤差の地図を作成した。

## 結果

- 毎時snapshotの読み戻し: 合格。float32化による最大速度ベクトルRMSEは`2.3343e-9 m/s`。
- 欠落時刻の直線補間: 不合格。
  - -11時間: 速度ベクトルRMSE `0.01626 m/s`、流速MAE `0.009742 m/s`。
  - -10時間: 速度ベクトルRMSE `0.01166 m/s`、流速MAE `0.005429 m/s`。
  - -9時間: 今回の基準内。

高精度の流速地図を優先するため、表示する全1時間時刻の直接計算snapshotを保持し、欠落時刻を前後の時刻から直線補間する代替経路は採用しない、という候補結論になった。

地図の全体像は似て見えるが、誤差地図では上流側と河口堰・分流周辺に局所差がある。画像は`docs/visuals/stage20-reference-s02-browser-comparison-decision.jpg`。

## まだ確認していないこと

- 異なる流量・潮位・堰条件の間を補間したときの精度
- 実測流速・水位との一致
- 当日の予報精度
- 実ブラウザでの処理時間（今回の速度測定はNode.jsであり、ブラウザ性能の主張には使わない）

次の判断Aは、この結果を採用して、低条件・高条件の基底から中間条件を推定し、別の直接solver結果を正解として比較するholdout試験の「計画だけ」を作る。Aを選んでも追加の物理計算、S03、残りsegment、公開接続、`main`反映は行わない。
