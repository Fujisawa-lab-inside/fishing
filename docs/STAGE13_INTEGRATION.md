# v4.8統一水面・有限体積入力層の統合方針

## 目的

水面判定，ヒートマップ生成領域，流体計算領域を，`waterDomain.rows` が表す唯一の正解水面へ統一する．

## 今回追加する基盤

- `data/onga_unified_water_manifest_r3.json`
  - 承認済み水面680,633 px
  - r2へ芦屋橋下の水面842 pxだけを追加し，既存水面は削除しない
  - G1・G2・G3の微小ノイズ445 pxを除外済み
  - 橋，河口堰，M/N/O/G，魚道の位置情報を保持
- `config/onga_recommended_model_v1.json`
  - 魚道固定流量，M潮位時系列，N/O/G流量境界，河口堰一様開度を仮採用
  - 実値は未設定
- `config/onga_physical_input_schema_v1.json`
  - 単位，符号規約，入力検証規則を定義
  - 実値未承認のためsolver接続は無効
- `onga_stage13_runtime.js`
  - 上記3ファイルを読み込み，水面マスクを一度だけ復号する
  - `window.OngaUnifiedAuthority.water.contains(x, y)` を全処理の共通判定関数として提供する

## 統合規則

1. 水面判定は `OngaUnifiedAuthority.water.contains` だけを参照する．
2. ヒートマップ候補点は，上記判定が真の点だけを生成する．
3. 流体格子・有限体積セルは，同じ水面マスクから生成したデータだけを使用する．
4. `water_polygon`，評価点群外縁，legacy shorelineを正解領域として再解釈しない．これらは表示・診断用に限定する．
5. 橋は水面を遮断しない．河口堰だけを内部interfaceとして扱う．
6. 魚道は上流側−Q，下流側＋Qの保存的内部移送とする．
7. 実潮位・流量・開度が承認されるまで，物理solverを本番出力へ接続しない．

## 次の実装段階

PC版・スマホ版ラッパーへ `onga_stage13_runtime.js` を読み込み，既存の水面判定，ホットスポット生成，流体計算入口を共通authorityへ差し替える．差し替え後，以下を自動検査する．

- 3処理の参照画素数が680,633 pxで一致する
- 水面外ヒートマップ点が0
- 水面外流体セルが0
- 河口堰閉鎖時の残存横断面が0
- PC版とスマホ版で水面マスクのハッシュが一致する
