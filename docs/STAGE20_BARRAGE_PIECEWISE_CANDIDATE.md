# Stage 20 河口堰区間別補間候補

## 現在の位置づけ

この候補は、直接計算済みの河口堰開度 0%・50%・100%を固定点にし、0～50%と50～100%を別々に線形補間する、内部確認用のコード候補である。

- 追加の物理計算は行っていない。
- 既存の公開シミュレータ、36時間表示、`main`には接続していない。
- 25%・75%は補間結果であり、直接計算による物理精度の確認結果ではない。
- 対象は、既存結果と同じ境界入力、モデル時刻 -12～-8、5枚の毎時snapshot、5時間を通して一定の河口堰開度だけである。

## 補間方法

河口堰開度を `p`（0～1）とし、同じモデル時刻の直接計算結果を `F0`、`F50`、`F100` とする。

- 0～50%: `F(p) = (1 - 2p) F0 + 2p F50`
- 50～100%: `F(p) = (2 - 2p) F50 + (2p - 1) F100`
- 0%・50%・100%では、それぞれの保存済み直接計算値をfloat32化してpackした固定点をそのまま返す。

この構成により、candidate pack内では50%の固定点と値の連続性が保証される。元のfloat64直接計算値からfloat32への最大速度ベクトル量子化誤差は `3.0445e-8 m/s` 以下である。一方、50%の左右で変化率は一致せず、既存のholdout結果から大きな折れがあることが分かっている。

## コード検証

`config/stage20_barrage_piecewise_validation_v1.json` に次を記録する。

- 0%・50%・100%のpacked固定点をbit単位で再現すること
- 25%・75%が、それぞれ隣接する固定点のfloat32中点と一致すること
- 同じ入力が同じ出力を返すこと
- 補間係数が常に非負で合計1になること
- NaN、無限値、範囲外開度、時刻補間、時刻外挿、欠落時刻を拒否すること
- 出力に非有限値と負水深がないこと
- solver、追加物理run、retry、S03、公開接続を行っていないこと

さらに `config/stage20_barrage_piecewise_source_pack_validation_v1.json` では、pack manifestの自己申告を使わず、証拠固定済みの15枚のfloat64 snapshotから全payloadを再構成し、candidate packとbyte単位で一致することを確認する。

packは15枚の保存済みfloat64 snapshotをfloat32化したもので、packと5時間分の出力を合わせた作業メモリは約11.49 MiBである。計測値は開発環境でのコード検証値であり、利用者端末すべてのブラウザ速度を保証するものではない。

## 使用できない範囲

- 過去12時間・未来24時間の36時間サービス
- モデル時刻 -12～-8以外
- 5時間の途中で河口堰開度が変わる条件
- 元の直接計算と異なる潮位、河川流量、雨条件
- 25%・75%の直接計算精度を確認済みとする表現
- 実測再現、日々の予報、釣果向上の精度を確認済みとする表現

## 採用結果

判断画像は `docs/results/stage20-barrage-piecewise-candidate-v1/maps/piecewise-candidate-overview.jpg` である。河口全域、河口堰、曲川・遠賀川合流点、魚道について、0%・25%・50%・75%・100%と、50%前後の変化率の折れを同時に示す。

モデル内相対 -9時間の速度ベクトル刻み差RMSEは、河口全域 `0.02766 m/s`、河口堰 `0.02255 m/s`、合流点 `0.00418 m/s`、魚道 `0.02238 m/s` だった。これは新しい直接計算結果ではなく、既存の0%・50%・100%結果から求めた50%前後の折れである。

2026-07-17、次のA案が選択された。

> A：50%を固定できる利点を優先し、変化率の折れを明示した内部コード候補として保持する

この選択により、候補コード、固定済み証拠、判断画像を作業ブランチ上の独立した内部候補として保持する。採用記録は `config/stage20_barrage_piecewise_candidate_adoption_v1.json` に固定する。

この選択は、実行時コードへの組み込み、公開シミュレータへの接続、追加の物理計算、retry、reference S03、36時間サービス化、`main`反映を許可しない。これらを行う場合は別の判断が必要である。
